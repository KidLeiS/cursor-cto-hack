# Authentication and security

## Access model

- Supabase Auth provides passwordless email magic links.
- Only `eric@aimalcolm.com` may create an authenticated account or access the
  project.
- Every other email submitted through `/login` is stored in
  `waitlist_entries` with `pending` status.
- Middleware and route-level guards protect pages, APIs, AI/transcription cost
  endpoints, and server actions.
- PostgreSQL RLS denies anonymous project access and checks project membership
  for authenticated access.
- Security-definer mutation RPCs are executable only by `service_role`.

## Required Vercel variables

```text
SB_URL
SB_PK
SB_SERVICE_ROLE_KEY
NEXT_PUBLIC_PROJECT_SLUG
NEXT_PUBLIC_SITE_URL
AUTH_RATE_LIMIT_SECRET
```

`SB_SERVICE_ROLE_KEY` and `AUTH_RATE_LIMIT_SECRET` are server-only. Never use a
`NEXT_PUBLIC_` prefix for either. Set `NEXT_PUBLIC_SITE_URL` to the canonical
production origin so magic links return to `/auth/callback`.

In Supabase Auth URL Configuration, add the production `/auth/callback` URL and
any preview callback URLs used for testing. Keep email confirmation enabled.

## MCP credentials

MCP keys are generated by an authenticated user from **Connect MCP**. The
plaintext has 256 bits of randomness and is shown once. PostgreSQL stores only:

- a non-secret lookup prefix;
- a SHA-256 hash;
- owner and project IDs;
- scopes, use timestamps, expiry, and revocation metadata.

Every MCP call validates key hash, project membership, scope, expiry,
revocation, and a 60-call-per-minute limit. Tool calls write success/failure
audit events without recording arguments or secrets.

## Deployment order

1. Add the required Vercel variables.
2. Merge and let CI apply migration `008_auth_waitlist_security.sql`.
3. Vercel production builds wait for an anonymous RLS-denial probe before
   promotion, preventing deployment ahead of the migration.
4. Sign in as `eric@aimalcolm.com`.
5. Generate a new personal MCP key. Shared `MCP_API_KEY` values are no longer
   accepted.

## Key rotation

Revoke a key in **Connect MCP**, generate a replacement, then update Cursor.
Revocation is effective on the next stateless MCP request.
